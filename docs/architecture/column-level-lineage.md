# ADR: Column-Level Lineage Architecture

**Date**: 2026-03-18
**POC**: @pingu
**TL;DR**: Extend table-level lineage to column-level by combining SQLModel field metadata (`source_column`) with optional SQL AST analysis (sqlglot). Two modes: declarative (metadata-driven, zero dependencies) and automatic (SQL parsing, opt-in). Column lineage surfaces through OpenLineage's `SchemaDatasetFacet` with `inputFields`/`outputFields`.

## Status

Proposed

## Context

sqldim currently tracks lineage at table granularity. `LineageEvent` records which tables were read and written, but not which columns produced which columns. This is sufficient for pipeline-level orchestration but insufficient for:

- **Impact analysis**: "Which reports break if I remove `customer.email`?"
- **Data contract validation**: "Is this silver column derived from the expected bronze column?"
- **Regulatory compliance**: "What source fields contributed to this PII-derived column?"
- **Transformation debugging**: "Why does `total_amount` differ between silver and gold?"

### Current State

| Level | Status | Mechanism |
|-------|--------|-----------|
| Table-level | ✅ Implemented | `LineageEvent.inputs/outputs` as `DatasetRef` |
| Column-level | ❌ Missing | No field-to-field mapping |
| Transformation-level | ❌ Missing | No SQL AST analysis |
| OpenLineage emission | ✅ Implemented | `OpenLineageEmitter` translates to `RunEvent` |

### OpenLineage's Column Lineage Model

OpenLineage supports column-level lineage via `SchemaDatasetFacet`:

```json
{
  "facets": {
    "schema": {
      "fields": [
        {"name": "customer_name", "inputFields": [
          {"namespace": "bronze", "name": "raw_name", "transform": "UPPER(TRIM(...))"}
        ]}
      ]
    }
  }
}
```

This is the target format — sqldim needs to produce these field mappings.

## Decision

Two-mode architecture: **declarative** (always available) and **automatic** (opt-in with sqlglot).

### Mode 1: Declarative Column Lineage (Metadata-Driven)

Extend `Field()` with a `source_column` parameter that declares the upstream column:

```python
class CustomerSilver(DimensionModel, SCD2Mixin, table=True):
    __natural_key__ = ["code"]
    code: str = Field(source_column="raw_code")           # Direct mapping
    name: str = Field(source_column="raw_name")           # Direct mapping
    region: str = Field(source_column="raw_region")       # Direct mapping
    email_clean: str = Field(source_column="raw_email")   # Implies transformation
```

The `source_column` metadata is stored in `column.info["source_column"]` alongside existing dimensional metadata. No SQL parsing needed — the modeler declares the mapping.

**Advantages**: Zero dependencies, works for all backends, easy to understand, covers 80% of cases (most ETL is column-to-column with simple transforms).

**Limitations**: Doesn't capture multi-column derivations (e.g., `full_name = first_name || ' ' || last_name`) or complex SQL expressions.

### Mode 2: Automatic Column Lineage (SQL AST Analysis)

For multi-column derivations and complex SQL, use sqlglot to parse the DuckDB SQL generated by lazy processors:

```python
# Opt-in at the pipeline level
pipeline = Pipeline(
    sources=[bronze_customers],
    loader=LazyType2Processor(model, sink, track_columns=["name", "region"]),
    column_lineage="auto",  # Enable sqlglot analysis
)
```

The analyzer:
1. Captures the SQL that the lazy processor generates
2. Parses it with sqlglot to extract `SELECT ... AS target_col FROM ...` mappings
3. Resolves column references back to source `DatasetRef` names
4. Produces `ColumnLineageFacet` entries with the parsed transform expression

**Advantages**: Handles multi-column derivations, captures actual SQL expressions, no manual declaration.

**Limitations**: Requires sqlglot dependency, DuckDB-specific SQL dialect may need patches, complex CTEs may not fully resolve.

### ColumnLineageFacet Data Model

```python
@dataclass
class ColumnLineageEntry:
    output_column: str
    input_columns: list[str]
    transform_description: str  # Human-readable, e.g., "UPPER(TRIM(raw_name))"
    transform_sql: Optional[str] = None  # Raw SQL fragment (Mode 2 only)
    confidence: str = "declared"  # "declared" (Mode 1) or "parsed" (Mode 2)

@dataclass
class ColumnLineageFacet:
    entries: list[ColumnLineageEntry]
```

### Integration with Existing Lineage Pipeline

Column lineage attaches to the existing `DatasetRef.facets` dict:

```python
# In DimensionalLoader.run() after load completes
output_ref = DatasetRef(
    namespace="silver",
    name="customer_dim",
    facets={
        "columnLineage": ColumnLineageFacet(
            entries=[
                ColumnLineageEntry(
                    output_column="name",
                    input_columns=["raw_name"],
                    transform_description="Direct mapping",
                    confidence="declared",
                ),
            ]
        ),
    },
)
```

`OpenLineageEmitter` serializes this facet into the OpenLineage `SchemaDatasetFacet.fields[].inputFields` format.

### Multi-Source Aggregation

For columns derived from multiple sources (e.g., a fact joining two dimensions):

```python
# Declarative mode supports multiple sources via list
total_amount: float = Field(
    source_columns=["bronze.line_amount", "bronze.tax_amount"],
    transform_description="line_amount + tax_amount",
)
```

### Metadata-Only Fallback

Even without `source_column` declarations, sqldim can infer **structural** column lineage from the schema graph:

- Fact FK columns → mapped to dimension natural key columns (via `Field(dimension=...)`)
- SCD2 `checksum` → derived from all `track_columns`
- Bridge `weight` → implicit (always 1.0 default)

This "free" lineage requires zero user configuration and covers FK relationships automatically.

## Consequences

**Positive**
- Declarative mode covers 80% of cases with zero new dependencies
- OpenLineage integration is incremental — facet extension, not protocol change
- Structural FK lineage is automatic — no user action needed for fact-dimension relationships
- Paves the way for impact analysis tooling ("which models reference this column?")

**Neutral**
- `source_column` is a single string — multi-source requires `source_columns` (plural)
- sqlglot is an optional dependency — only imported when `column_lineage="auto"`
- `transform_description` is human-readable but not machine-verifiable

**Negative**
- Declarative mode can drift from actual SQL — if the SQL changes but `source_column` isn't updated, lineage is wrong
- sqlglot parsing of DuckDB SQL is not 100% reliable — window functions, CTEs, and DuckDB-specific syntax may fail
- Column lineage increases `LineageEvent` payload size significantly for wide tables
- No backward lineage ("which columns does X feed into?") — only forward lineage ("where does X come from?")

## Future Work

| Item | Description |
|------|-------------|
| Backward lineage index | In-memory graph of column → downstream columns for impact analysis |
| Lineage drift detection | Compare declarative `source_column` against parsed SQL and warn on mismatches |

| Lineage visualization | ERD-style column lineage graph (extension of schema graph rendering) |
| Incremental lineage | Only compute column lineage for changed columns, not full table |
